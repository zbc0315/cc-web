# CC Web — 产品定位分析与改进建议

> 基于 v1.5.48 全量代码 review，2026-03-26

---

## 一、这个程序是什么

CC Web 是一个**自托管的 Claude Code 浏览器前端**。

它不是一个新的 AI 产品，也不是 Claude API 的封装——它是把 Anthropic 官方 CLI 工具 `claude`（即 Claude Code）搬到浏览器里的"壳"。底层完整运行的是用户本地安装的真实 `claude` 二进制，CC Web 只是在它外面包了一层：

- **PTY 层**：用 `node-pty` spawn 一个真实的 shell，在里面运行 `claude`，把输入/输出通过 WebSocket 转发到浏览器的 xterm.js 终端
- **会话层**：读取 Claude Code 自己写的 JSONL 会话文件（`~/.claude/projects/`），解析后以气泡对话界面展示
- **管理层**：项目管理、文件浏览、快捷命令、云备份、多用户隔离

### 核心价值

| 场景 | CC Web 解决的问题 |
|------|-----------------|
| 在远程服务器上运行 Claude Code | 不需要 SSH + tmux，浏览器直接访问 |
| 多项目并行 | Dashboard 统一管理，会话持久化，随时切换 |
| 团队协作 | 多用户账号 + 项目分享（view/edit 权限） |
| 不在电脑旁 | 手机/平板浏览器也能看到 Claude 的输出 |
| 会话记录 | 对话历史以结构化视图展示，而非滚动终端 |

## 二、目标用户画像

**核心用户 A：重度 Claude Code 用户**
- 每天打开多个项目，需要项目管理和快速切换
- 想在浏览器里而不是终端里操作
- 希望看到干净的对话界面而不是 ANSI 转义序列

**核心用户 B：远程开发者**
- Claude Code 跑在服务器（Mac Mini、VPS、家庭 NAS）上
- 需要从任何设备访问，包括 iPad、公司电脑
- 需要 LAN/公网访问控制

**边缘用户 C：小团队 / 家庭共享**
- 一台机器多个人用，各自有独立工作区
- 需要项目分享和权限控制

## 四、改进建议

### 4.1 核心体验缺口

#### A. 任务完成通知（高价值，实现不难）

**问题**：Claude Code 处理耗时任务时（编译、重构、生成大量文件），用户必须盯着终端。没有任何通知机制。

**建议**：
- 当检测到 Claude Code "Stop" hook 触发（即一轮对话结束），发送浏览器 Push Notification
- 配置：可设定"空闲超过 X 秒才通知"（避免快速问答也通知）
- 扩展：支持 Webhook（POST 到用户指定 URL），便于接入 Telegram/企业微信/Slack

```
实现路径：
Stop hook → POST /api/hooks → SessionManager.clearSemanticStatus()
在这里新增 → 推送 Web Push Notification 或 Webhook
```

---

#### B. 终端内搜索（高频需求，用户会主动找）

**问题**：xterm.js 支持 SearchAddon，但当前未接入。5MB 滚动缓冲有大量内容，无法搜索。

**建议**：
- 接入 `@xterm/addon-search`
- Ctrl+F（或 Cmd+F）触发搜索框，支持正则、大小写敏感选项
- 高亮匹配项，上/下翻查

---

#### C. 对话历史全局搜索（跨项目、跨会话）

**问题**：当前 Session 面板只能查看单个项目的历史，且只能按时间排列，无法按内容搜索。

**建议**：
- `GET /api/sessions/search?q=<keyword>&projectId=...`
- 后端遍历所有项目的 JSONL，提取文本匹配的片段
- 前端：Dashboard 顶部增加全局搜索入口，结果展示匹配的消息上下文 + 跳转链接

---

#### D. Git 状态面板（Claude Code 最常做的事是改代码）

**问题**：Claude Code 的核心用途是修改代码。用户每次要看改了什么，都要打开另一个终端/编辑器。

**建议**：
- 项目页右侧面板增加"Git"标签（与"快捷命令 / 历史 / 图谱"并列）
- 展示：`git status`（变更文件列表）+ `git diff`（行级 diff）
- 操作：支持直接在 UI 里 `git add` / `git commit`（发送指令到 PTY）
- 不用解析 git 输出，直接调用 `simple-git` 库

---

#### E. 移动端适配（当前完全不可用）

**问题**：三栏布局在手机上会崩。Panel 切换全靠 hover/右键。xterm.js 虚拟键盘体验差。

**建议**（分阶段）：
- **最小可用**：窄屏下折叠为单栏，FileTree/Terminal/RightPanel 以底部 Tab 切换
- **进阶**：对话历史标签页在手机上可以以"只读模式"展示（不需要终端），适合碎片时间查看进度
- **虚拟键盘**：为手机端增加常用符号快捷键盘（`Tab` `Esc` `Ctrl+C` `方向键`），解决 xterm.js 移动端痛点

---

### 4.2 项目管理增强

#### F. 项目分组 / 标签

**问题**：项目多了（>10个）Dashboard 变得混乱，没有分类手段。

**建议**：
- 项目支持打 `tags`（字符串数组）
- Dashboard 顶部增加标签过滤器
- 可选：支持拖拽分组 / 折叠

---

#### G. 项目使用统计面板

**问题**：用户不知道每个项目用了多少 token、多少时间、改了多少文件。

**建议**：
- 利用现有 JSONL + backup-state 数据，派生统计：
  - 本周/本月对话轮次
  - 近 N 次对话的平均时长
  - 文件变更数（需要 git 或 snapshot diff）
- 在项目卡片 hover 展示简要数字，点击展开详细 Timeline

---

#### H. 任务看板（对接 Claude Code Todo）

**问题**：Claude Code 有内置 TodoWrite/TodoRead 工具，写入项目的 `.todos` 或对话状态。但 CC Web 完全没有展示这部分。

**建议**：
- 监听 PostToolUse hook 中 `TodoWrite` 工具调用，解析 todo 列表
- 在 RightPanel 增加"任务"标签，实时展示 todo 状态（pending / in_progress / completed）
- 用户可以手动勾选/取消，直接发指令到 PTY 更新 todo

---

### 4.3 协作与分享

#### I. 会话分享链接

**问题**：当前项目分享需要注册账号，且是全项目级别的。有时用户只想把某次对话的截图/链接发给同事。

**建议**：
- `POST /api/sessions/:id/share` → 生成一个 token 化的公开只读链接（有效期可配置）
- 访问链接：无需登录，只读展示该 session 的气泡对话
- 安全：链接携带 signed token，服务端校验，不暴露文件路径

---

#### J. 多用户实时协同终端（进阶）

**问题**：当前分享是"你看你的，我看我的"——即使 edit 权限用户，双方输入也不互相可见（因为 broadcast 只在 rawBroadcast 时发送）。

**建议**：
- `broadcastFn` 改为多播：维护每个项目的 WS 客户端集合，PTY output 广播给所有订阅者
- 任何有 edit 权限的用户的 `terminal_input` 都写入 PTY
- 展示：终端顶部显示"当前在线"用户头像（类 Google Docs 游标）

---

### 4.4 可靠性与运维

#### K. 结构化错误上报

**问题**：后端异常目前只打印到 `console.error`，前端也只有 toast。没有集中的错误追踪。

**建议**：
- 接入轻量级本地错误聚合（如写入 `~/.ccweb/error.log`，保留最新 1000 条）
- `GET /api/admin/errors`：管理员可查看近期错误日志
- 前端 ErrorBoundary 增加"报告此问题"按钮（打开 GitHub Issue 模板）

---

#### L. 健康检查 & 进程监控

**问题**：PTY 进程挂起（不退出但不响应）时，当前只能靠用户手动 stop/start。

**建议**：
- 每个 PTY 维护一个"最后输出时间"（已有 `lastActivityAt`）
- 配置"空闲超时"：如果 PTY 超过 N 分钟无输出且状态为 running，弹出提醒（不自动 kill）
- 可选：管理员面板展示所有 PTY 的 PID、内存占用、运行时长

---

#### M. 备份恢复 UI

**问题**：备份已实现，但没有恢复功能的 UI。用户必须手动下载文件再上传。

**建议**：
- Settings 页 BackupHistory 增加"恢复"操作：选择某次备份版本 → 列出变更文件 → 确认后下载到本地
- 简化版：直接提供"打开云端目录"链接（跳转到 Google Drive / OneDrive 对应文件夹）

---

### 4.5 开发者体验

#### N. API 文档 / OpenAPI Spec

**问题**：CC Web 有丰富的 REST API，但没有文档。第三方集成（脚本、自动化工具）需要翻看源码。

**建议**：
- 添加 `/api/docs`（swagger-ui-express）自动生成 OpenAPI 文档
- 只需在路由上加 JSDoc 注释，自动提取

---

#### O. 插件 / 扩展机制

**问题**：当前功能全部硬编码在后端路由和前端组件里。加新功能需要改核心代码。

**建议**：
- 定义简单插件接口：后端插件注册额外路由，前端插件注册额外 RightPanel 标签
- 插件放在 `~/.ccweb/plugins/` 目录，启动时动态加载
- 这允许社区开发功能而不 fork 主仓库

## 五、优先级建议

| 优先级 | 功能 | 理由 |
|--------|------|------|
| P0 | **任务完成通知（F+Webhook）** | 解决"盯着屏幕等"的核心痛点，实现成本低 |
| P0 | **终端内搜索** | xterm.js 已有插件，接入工作量小，用户高频需求 |
| P1 | **对话历史全局搜索** | 积累会话后强需求，后端只需遍历 JSONL |
| P1 | **Git 状态面板** | Claude Code 核心场景是改代码，与产品目的高度契合 |
| P1 | **移动端最小适配** | 远程访问场景必然有手机需求，至少要能看对话历史 |
| P2 | **项目标签/分组** | 用户项目多了自然需要，但不影响核心功能 |
| P2 | **会话分享链接** | 低成本提升协作体验 |
| P2 | **Todo 任务看板** | 深度利用 Claude Code 原生工具，差异化特色 |
| P3 | **协同终端** | 技术复杂度高，用户量达到一定规模再考虑 |
| P3 | **插件机制** | 生态建设，适合产品成熟后再推进 |

## 六、产品定位的一个核心矛盾

CC Web 同时服务两类场景：

1. **个人用户**：一台机器、一个人、方便管理多项目
2. **团队/远程**：多用户、权限控制、协作

这两类场景对 UI 复杂度的预期完全不同。个人用户希望"简单，直接开干"；团队用户希望"精细控制，能审计"。

**建议**：在 `ccweb start` 时增加"个人模式 / 团队模式"选项，个人模式隐藏分享、多用户、备份等复杂功能，提供更干净的 UI。团队模式开启全部功能。这不需要两套代码，只需一个配置开关。
