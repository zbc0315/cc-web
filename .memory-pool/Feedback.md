# Memory Pool 系统反馈

## 2026-04-04 — AI 实际使用反馈

**优势：**
1. 类型分层 + B₀ 赋值有效：feedback(B₀=8-10) 始终在顶部，reference(B₀=2-4) 自然沉底，无需手动排序
2. 无淘汰设计正确：所有信息持久存在，只是读取优先级不同，避免误删
3. links 机制有潜力：关联球在检索相关上下文时有用
4. hardness 区分了"可变项目状态"(4-6)和"不可变规则"(8-10)

**劣势：**
1. 手动簿记成本高：每次更新需操作 ball文件 + index.json + state.json，≥6次文件操作，占用大量 token 和时间
2. 浮力计算无自动化：λ^(t-t_last) 衰减需要手动计算或脚本，实际操作中容易出错或跳过
3. t（轮次）在跨会话场景下语义模糊：每次新对话 t 不自动递增，需要读取 state.json 并手动 +1
4. 与 auto-memory 功能冗余：Claude Code 自带的 .claude/projects/memory/ 系统更轻量，实际决策时更多参考 auto-memory 而非记忆池
5. Fusion/Fission 在当前规模（16球）下从未触发：co-occurrence≥4 的阈值在实际使用中几乎不可能达到
6. 记忆池未被用于实时决策：回答问题时并不真正依据浮力排序来选择上下文，而是读取所有活跃球后自行判断相关性

**建议改进方向：**
- 自动化轮次递增和浮力衰减计算（hooks 或 session-start 脚本）
- 减少每次操作的文件接触数（合并 index 和 state？）
- 降低 fusion 阈值或改为基于语义相似度而非共现计数
- 明确记忆池与 auto-memory 的分工边界
